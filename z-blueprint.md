# z-blueprint.md

A reusable, config-driven CDC streaming ETL framework on AWS.
Define tables, schemas, DQ rules, transforms, and Deequ checks in a single `tables.yaml` — the framework validates, compiles, deploys, and runs the entire pipeline. No code changes for new use cases.

```
tables.yaml ──→ Config Validator ──→ Config Compiler ──→ deploy.sh ──→ Isolated AWS Stack
                                          |
                    ┌─────────────────────┼─────────────────────┐
                    ▼                     ▼                     ▼
              DMS Mappings          RDS DDL              CFn Parameters
              (JSON)                (SQL)                (JSON)
```

```
RDS PostgreSQL ──→ AWS DMS ──→ Amazon MSK (Kafka) ──→ AWS Glue Streaming ──→ Delta Lake (S3)
     |                |               |                       |                    |
  INSERTs        CDC capture     per-table topics      7-step pipeline        SCD2 history
  UPDATEs        JSON envelope   SASL/SCRAM auth       DQ + transforms        + quarantine
  DELETEs        port 9096                             + Deequ metrics         + Athena
                                                       + SNS alerts
```

## Quick Reference — "I want to…"

| I want to… | Go to… |
|------------|--------|
| Deploy a new use case | Write `examples/{name}/config/tables.yaml` + `scripts/data_generator.py`, run `deploy.sh --use-case {name}` |
| Add a new table to a use case | Edit `examples/{use-case}/config/tables.yaml` → add table entry. Redeploy. Config compiler generates DDL + DMS mappings automatically. |
| Add a new DQ rule type | `src/glue_streaming_job.py` → `DQRegistry.__init__()` → add to `self._rules` dict + implement method |
| Add a new transform type | `src/glue_streaming_job.py` → `TransformRegistry.__init__()` → add to `self._transforms` dict + implement method |
| Add a new Deequ metric | `src/deequ_analyzer.py` → `_build_analyzers()` → add `elif` block |
| Change DQ rules for a table | Edit `tables.yaml` → `dq_rules` section (no code changes) |
| Change transforms for a table | Edit `tables.yaml` → `transforms` section (no code changes) |
| Deploy a stack | `./scripts/deploy.sh --stack-name X --use-case Y --region Z` then `./scripts/post-deploy.sh --stack-name X --use-case Y --region Z` |
| Tear everything down | `./scripts/teardown.sh --stack-name X --region Z --force` |
| Check quarantined records | S3: `s3://{stack}-quarantine-{account}/run_date={YYYY-MM-DDTHH:00:00}/{table}/` — Parquet with `_failed_rule` and `_quarantined_at` columns |
| Get DQ failure alerts | Set `SNS_NOTIFICATION_EMAILS` before deploy. Alerts include table, count, failed rules, S3 path. |
| Query processed data | Athena → `SELECT * FROM "{stack}_db"."{table}" LIMIT 10;` |
| See Deequ metrics | Athena → `SELECT * FROM "{stack}_db"."dq_metrics" ORDER BY timestamp DESC` |
| Review generated artifacts before deploy | `python3.10 -m src.config_compiler --config examples/{use-case}/config/tables.yaml --output build/` |
| Understand CloudFormation resources | `cloudformation/streaming-etl.yaml` — single parameterized template |

## The Story of a Row

**1. A row changes in RDS PostgreSQL**

Tables are created automatically from `tables.yaml` by the config compiler (generates DDL) and the SQL Runner Lambda (executes it). The database has `wal_level=logical` enabled for CDC.

**2. DMS captures the change**

AWS DMS watches the PostgreSQL WAL. It wraps each changed row in a JSON envelope and sends it to a per-table Kafka topic:

```json
{
  "data": { "id": 1, "speed_kmh": 400.0, "latitude": 40.7128 },
  "metadata": { "operation": "insert", "schema-name": "public", "table-name": "vehicle_telemetry" }
}
```

Topic naming: `cdc-{table_name}` (auto-generated from config if omitted).
Table-to-topic mapping: generated by config compiler into `build/dms_table_mappings.json`.

**3. Kafka holds the message**

Amazon MSK (Kafka 3.5.1) with SASL/SCRAM-SHA-512 auth on port 9096. Credentials in Secrets Manager as `AmazonMSK_{stack}_scram`, encrypted with a dedicated KMS key.

**4. Glue picks it up**

`src/glue_streaming_job.py` → `main()`:

```
main()
  → Reads CONFIG_PATH and SNS_TOPIC_ARN from Glue job args
  → load_config_from_s3()        # Loads tables.yaml, resolves ${STACK_NAME} and ${AWS_ACCOUNT_ID}
  │   → Resolves MSK bootstrap servers via MSK API
  → get_kafka_password()          # Reads SCRAM password from Secrets Manager
  → create_deequ_analyzer()       # Initializes DeequAnalyzer with S3 state path
  → For each non-internal table in config:
      → start_stream_for_table()
          → Builds Kafka SASL/SCRAM connection options
          → spark.readStream.format("kafka").load()
          → writeStream.foreachBatch(process_batch).start()
```

One streaming query per table, all running in parallel. Each fires `process_batch()` per trigger interval (default 15 seconds).

**5. The 7-step micro-batch pipeline**

Inside `process_batch()`:

```
process_batch(batch_df, batch_id, table_config, config, spark, deequ_analyzer)
  │
  ├─ Step 1: parse_dms_messages()
  │    Parses DMS JSON envelope. Extracts data columns + maps operation.
  │    Filters out control messages (all-null rows).
  │
  ├─ Step 2: deequ_analyzer.analyze_batch()
  │    Runs Deequ analyzers (completeness, uniqueness, compliance, size).
  │    Persists state to S3. Writes metrics to Delta Lake.
  │    DOES NOT FILTER ROWS — only measures and records.
  │
  ├─ Step 3: apply_dq_rules()
  │    Iterates over table's dq_rules from config.
  │    DQRegistry.apply() returns (passed_df, failed_df) for each rule.
  │    Failed records (severity=error) tagged with _failed_rule.
  │    → write_to_quarantine() writes to s3://{stack}-quarantine-{account}/run_date={YYYY-MM-DDTHH:00:00}/{table}/
  │    → send_quarantine_notification() publishes to SNS (table, count, failed rules, S3 path)
  │    Only passed records continue.
  │
  ├─ Step 4: apply_transforms()
  │    Iterates over table's transforms from config, sorted by `order`.
  │    TransformRegistry.apply() modifies columns in-place.
  │
  ├─ Step 5: add_scd2_columns()
  │    Adds: _effective_from=now, _effective_to=null, _is_current=true, _ingested_at=now
  │
  ├─ Step 6: deduplicate_by_pk()
  │    Window function over primary_key, ordered by _ingested_at desc.
  │    Keeps only the latest record per PK within this batch.
  │
  └─ Step 7: Delta MERGE (SCD Type 2) + Symlink Manifest
       If Delta table doesn't exist → write as new table.
       If it exists:
         a) MERGE: close old record (_is_current=false, _effective_to=now)
         b) Append new version (_is_current=true)
       After every write: delta_table.generate("symlink_format_manifest")
       for Athena compatibility.
```

**6. Data lands in S3**

Three S3 buckets (`{stack}-{purpose}-{account}`):
- `delta` — Delta Lake tables + `_symlink_format_manifest/` + `/dq_metrics/` + `/deequ-state/`
- `quarantine` — Failed DQ records as Parquet, partitioned: `run_date={YYYY-MM-DDTHH:00:00}/{table}/`
- `assets` — Scripts, config, wheels, JARs, checkpoints, Lambda layers

**7. Athena queries it**

The `{stack}-athena-table-creator` Lambda reads `tables.yaml` from S3, builds column definitions (schema + SCD2 columns), and creates Athena external tables using `SymlinkTextInputFormat` pointing to `_symlink_format_manifest/` directories. All identifiers are backtick-quoted for hyphenated stack names.

## The Config File

`tables.yaml` is the single file that drives the entire pipeline. Located at `examples/{use-case}/config/tables.yaml`. The Glue job loads it from S3 and resolves placeholders at runtime:
- `${STACK_NAME}` — derived from the S3 bucket name
- `${AWS_ACCOUNT_ID}` — from STS GetCallerIdentity

**Config → Code mapping:**

```
tables[].schema          →  build_schema_from_config()    →  Spark StructType for parsing
                         →  ConfigCompiler.compile_rds_ddl()  →  PostgreSQL CREATE TABLE
                         →  Athena Table Creator Lambda   →  Athena external table columns
tables[].dq_rules        →  DQRegistry.apply()            →  (passed_df, failed_df)
tables[].deequ_checks    →  DeequAnalyzer.analyze_batch() →  metrics DataFrame → Delta
tables[].transforms      →  TransformRegistry.apply()     →  modified DataFrame
tables[].primary_key     →  deduplicate_by_pk()           →  Window(partitionBy=pk)
                         →  Delta MERGE condition          →  "target.{pk} = source.{pk}"
tables[].topic           →  ConfigCompiler.compile_dms_table_mappings()  →  DMS rules
                         →  Kafka Admin Lambda             →  topic creation
tables[].foreign_keys    →  ConfigCompiler.compile_rds_ddl()  →  FOREIGN KEY clauses
tables[].indexes         →  ConfigCompiler.compile_rds_ddl()  →  CREATE INDEX statements
settings.trigger_interval→  trigger(processingTime=...)
settings.quarantine_path →  write_to_quarantine() output
kafka.bootstrap_servers  →  "PLACEHOLDER" → resolved at runtime via MSK API
source_database.schema_name → ConfigCompiler → DMS object-locator schema-name
```

**DQ rule types (6):** `range`, `allowed_values`, `regex`, `not_null`, `unique`, `length`
**Transform types (8):** `trim`, `lower`, `upper`, `round`, `mask_pii`, `cast`, `default_value`, `rename`
**Deequ metrics (4):** `completeness`, `uniqueness`, `compliance`, `size`

All registries have `register()` methods for adding custom rules/transforms (plugin-ready for v2).

## Deployment

Two-phase deployment. `deploy.sh` builds infrastructure, `post-deploy.sh` sets up the application.

```
STEP 1: ./scripts/deploy.sh --stack-name X --use-case Y --region Z
====================================================================

  validate_use_case()           ←── Checks examples/{use-case}/config/tables.yaml exists
       |
  validate_config()             ←── python3.10 -m src.config_validator --config ...
       |
  compile_config()              ←── python3.10 -m src.config_compiler --config ... --output build/
       |                             Generates: dms_table_mappings.json, create_tables.sql, cfn_parameters.json
       v
  deploy_cloudformation_stack() ←── cloudformation/streaming-etl.yaml
       |                             Creates: VPC, RDS, MSK, DMS, Glue, S3, Lambdas, SNS, IAM, KMS
       |                             Takes ~30-45 min (MSK is the bottleneck)
       v
  upload_assets_to_s3()
       |  Uploads: glue_streaming_job.py, deequ_analyzer.py, tables.yaml, compiled artifacts
       |  Creates + publishes 3 Lambda layers: psycopg2, kafka-python, PyYAML
       |  Zips + deploys use-case data_generator.py to Data Generator Lambda
       v
  subscribe_notification_emails() ←── If SNS_NOTIFICATION_EMAILS is set
       |
  start_dms_replication()


STEP 2: ./scripts/post-deploy.sh --stack-name X --use-case Y --region Z
========================================================================

  create_rds_tables()       ←── Executes generated DDL via SQL Runner Lambda
       |
  create_kafka_topics()     ←── Invokes Kafka Admin Lambda with sync_from_config
       |
  start_dms_task()          ←── Tests connections, then starts replication
       |
  start_glue_job()          ←── Polls for STOPPED state (up to 3 min), then starts
       |
  create_athena_tables()    ←── Invokes Athena Table Creator Lambda (auto-creates DB)
       |
  generate_test_data()      ←── Invokes Data Generator Lambda: seed + burst
```

## Infrastructure

**CloudFormation resources** (single template: `cloudformation/streaming-etl.yaml`):

```
VPC + Subnets + NAT Gateway + Security Groups
  └→ RDS PostgreSQL (wal_level=logical)
       └→ MSK Cluster (SASL/SCRAM, 2 brokers)
            └→ MSKBootstrapServers (Custom Resource Lambda)
                 ├→ DMS (Replication Instance + Endpoints + Task)
                 ├→ Glue (Database + MSK Connection + Streaming Job)
                 ├→ SNS Topic (DQ Alerts)
                 └→ Lambdas:
                      ├─ SQL Runner (psycopg2 layer, VPC)
                      ├─ Kafka Admin (kafka-python + PyYAML layer)
                      ├─ Athena Table Creator (PyYAML layer)
                      ├─ Data Generator (psycopg2 layer, VPC — code deployed by deploy.sh)
                      └─ MSK Bootstrap (Custom Resource)
```

**S3 buckets** (`{stack}-{purpose}-{account}`):
- `assets` — scripts, config, wheels, JARs, layers, checkpoints
- `delta` — Delta Lake tables, symlink manifests, Deequ state, DQ metrics
- `quarantine` — failed DQ records (Parquet, partitioned by `run_date` + table)

**Secrets:**
- `AmazonMSK_{stack}_scram` — Kafka SASL/SCRAM creds, encrypted with dedicated KMS key

## Key Design Decisions

| What | Why |
|------|-----|
| Config-as-contract | Single `tables.yaml` drives everything: DDL, DMS mappings, Kafka topics, Glue pipeline, Athena tables |
| Config compiler | Eliminates manual maintenance of DMS mappings, DDL, CFn params — all generated from config |
| Isolated stacks | Each `--use-case` deployment is fully independent — no shared infrastructure |
| Two-phase Lambda deploy | CFn creates placeholder Lambda, deploy.sh pushes real code (avoids S3 chicken-and-egg) |
| Symlink manifests | `delta_table.generate("symlink_format_manifest")` after every write — works across all Athena versions/regions |
| Hourly quarantine partitions | `run_date=YYYY-MM-DDTHH:00:00/{table}/` — efficient querying without partition explosion |
| SNS DQ alerts | Glue publishes to SNS after quarantine write — optional email subscriptions via `SNS_NOTIFICATION_EMAILS` |
| Glue stop polling | Post-deploy polls for STOPPED state up to 3 min instead of fixed sleep — honest error reporting |
| RDS_HOST env var chain | Lambda reads `RDS_HOST` (from CFn) → Secrets Manager → `DB_HOST` → localhost — works in any deployment |
| Pre-downloaded wheels | Zero downloads during deploy — works in air-gapped environments |
| Backtick-quoted Athena SQL | Handles hyphenated stack names (e.g., `my-etl-stack_db`) |
| Schema types match DMS output | DMS converts: numeric→double, boolean→string, timestamp→string |
| SCD2 as two-step merge | Close old record, then append new — simpler than complex single merge |
| register() stubs | DQRegistry and TransformRegistry accept custom handlers — plugin-ready for v2 |

## Example Use Cases

| Use Case | Tables | Exercises |
|----------|--------|-----------|
| `vehicle-telemetry` | vehicles, drivers, vehicle_telemetry, deliveries, alerts (5) | Range DQ (GPS, speed, fuel), allowed_values, regex, not_null, PII masking, SCD2 updates/deletes |
| `healthcare-iot` | patients, vitals (2) | Clinical range DQ, PII masking (name, phone, address), foreign keys, high completeness thresholds, regex (MRN format) |

Both generators produce: seed data, DQ violations (~5-10%), SCD2 scenarios (20-30% updates/deletes), Deequ threshold triggers.
